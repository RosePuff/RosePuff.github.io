类加载器

作用：加载Class文件

  1. 虚拟机自带的加载器
  2. 启动类（根）加载器
  3. 扩展类加载器
  4. 应用程序加载器

双亲委派机制

类加载过程

  • 类记载器收到类记载的请求

  • 将这个请求向上委托给父类加载器去完成，一直向上，直到启动类加载器

  • 启动类加载器检查是否能加载到当前的这个类，能加载的话就结束，使用当前加载器，否则抛出异常，通知子类加载器进行加载

  • 重复第三步

  • null：Java调用不到（C、C++）

  • Java == C++：去掉繁琐的东西，指针之类的，最后调用的还是C++写的

public class String {
    /*
    双亲委派机制，安全性出发
    1、APP-->EXC-->BOOT（最终执行的地方）
    也就是说会层层向上找，如果有的话优先执行根加载器里面的，当没有触发下面的机制
    2、BOOT-->EXC-->APP（最终执行的地方）
    当根加载器没有这个类，会到扩展类加载器寻找，当没有的时候才回到当前类执行main()方法
     */
    public String toString() {
        return "Hello";
    }

    public static void main(String[] args) {
        String s = new String();
        System.out.println(s.getClass().getClassLoader());
        s.toString();

        new Thread().start();
    }

    /*
    1、类加载器收到类加载的请求
    2、将这个请求向上委派给父类去加载，一直向上，知道启动器类记载APP
    3、启动类加载器检查是否能加载到当前的这个类，能加载的话就结束，使用当前加载器，否则抛出异常，通知子类加载器进行加载
    4、重复第三步

    null：Java调用不到（C、C++）
    Java == C++：去掉繁琐的东西，指针之类的，最后调用的还是C++写的
     */
}

沙箱安全机制

内存溢出示例：

public class OutOfMemory {

    /*
    死循环，a()无限调用b()，同时b()无限调用a()，造成死循环，内存溢出
     */

    public static void main(String[] args) {
        new OutOfMemory().a();
    }

    public void a() {
        b();
    }

    public void b() {
        a();
    }
}

组成沙箱的基本组件：

  • 字节码校验器（bytecode verifier）：确保Java类文件遵循Java语言规范，这样可以帮助Java程序员实现内存保护。但是并不是所有的的类文件都会经过字节码校验阶段，比如核心类之类的类就不会经过字节码校验阶段，因为已经是无法修改的文件。

  • 类装载器（class loader）：其中类装载器在3个防霾你对Java沙箱起作用

    • 它防止恶意代码去干涉善意的代码。（双亲委派机制）
    • 它保护了被信任的类库边界。
    • 它将代码归入保护域，确定了代码可以进行哪些操作。

    虚拟机为不同的类加载器载入的类提供不同的命名空间，命名空间由一系列唯一的名称组成每一个被装载的类都会得到一个名字，这个命名空间是由Java虚拟机为每一个类装载维护的，相互之间不可见

    类装载器采用的是双亲委派机制模式。

    1. 从最内层的JVM自带类加载器开始加载，外层恶意同名的类得不到加载从而无法使用
    2. 由于严格通过包的路径来区分了访问域，外层恶意的类通过内置的代码也无法访问到内层类，破坏代码自然也就无法生效

  • 存取控制器（access controller）：存取控制器可以控制核心API堆操作系统的存取权限，而这个控制的策略的设定，可以由用户去指定

  • 安全管理器（security manager）：是核心API喝操作系统之间的主要接口，实现权限控制，比存取控制器优先级高。

  • 安全软件包（security package）：java.security下的类喝扩展包下的类，允许用户为自己的应用增加新的安全特性，包括：

    • 安全提供者
    • 消息摘要
    • 数字签名
    • 加密
    • 鉴别

Native关键字

Native：带了native 关键字段的，说明Java的作用域到达不了，会直接到调用底层语言的库

会进入本地方法栈

调用本地方法接口 JNI

JNI：扩展Java的使用，融合不同的编程语言为Java所用

最初想融合C AND C++

在内存区域中专门开辟了，本地方法栈，为了登记Native方法，会在最终执行的时候会加载本地方法栈，通过JNI接口

PC寄存器

程序计数器

每个线程都有一个程序计数器，是线程私有的，就是一个指针，指向方法区中的方法字节码（用来存储指向一条指令的过程，也即将要执行的指令代码），在执行迎请读取下一条指令的时候，是一个非常小的内存空间，几乎可以忽略不计

方法区

Method Area方法区

​ 方法区是被所有线程共享的、所有字段、字节码，以及一些特殊的方法，如构造函数，接口代码也是在此定义的，；

也就是说所有定义的方法的信息都是保存在该区域，，该区域属于共享区间

其中：静态变量、常量、类信息（构造方法、接口定义）、运行时的常量池存在方法区中， 但是，实际变量是存在堆内存中的，和方法区无关

也就是：static、final、Class（方法区存的东西）

栈

数据结构，一般使用队列做比较

程序 = 数据结构 + 算法（提升使用的），而不是程序 = 框架 + 业务逻辑（吃饭==必会的） ：SSM + SpringBoot + SpringCloud

先进后出，后进先出（桶的概念）

栈内存：主干程序的运行，生命周期和线程同步，线程结束，栈内存也就是释放了

对于栈来说不存在垃圾回收问题，一旦线程结束，栈就结束了

栈满了会抛出：StackOverflowError

      在JavaOne2010大会上，Oracle宣布JRockit的优秀特性将会在OpenJDK上实现。[2][3]当前，JRockit和HotSpot正在集成，目标是在JDK 8发布之时发布集成后的代码库。

  • IBM J9VM

    • J9大量为WebSphere等中间件做优化，它的RT版本主要也是为了WAS RT之流做的；另外就是做Specjbb的优化了

堆

Heap，一个JVM只有一个堆内存，堆内存的大小是可以调节的

类加载器读取了类文件后，一般会把类、方法、常量，保存的是引用类型的真实对象

新生区（真实的是幸存区动态的）

经历了一次GC没有回收的会进入到幸存区，0和1区不断交换，当超过设定GC次数会进入养老区

  • 伊甸园区（Eden Space）
  • 幸存0区
  • 幸存1区

理解

  • 类/对象：的诞生和成长的地方，甚至死亡
  • 伊甸园：所有的对象都是在伊甸园new出来的

经过研究表示，99%的对象都是临时对象

老年区

大多数情况不会被GC掉，当老年区的内存满了，会进行重GC，也就是Full GC

永久区

这个区域常驻内存的，用来存放JDK自身携带的Class对象。Interface元数据，存储的时Java运行时的环境，放在磁盘里的

这个区域不会进行GC，关闭JVM虚拟机就会释放这个区域的

  • JDK1.6之前：永久代，常量池时在方法区中。
  • JDK1.7：永久代，退化了，叫做去永久代，常量池在堆中
  • JDK1.8之后：无永久代，叫做元空间，常量池放在元空间中

元空间：逻辑上存在，物理上不存在

堆内存调优

MAT Jprofiler作用

  • 分析Dump内存文件，快速定位内存泄漏
  • 获得堆中的数据
  • 获得大的对象

JVM调优参数

  • -Xms初始化内存分配大小，默认（1/64）
  • -Xmx设置最大分配内存，默认（1/4）
  • -XX:+PrintGCDetails 打印GC垃圾回收信息
  • -Xms1m -Xmx8m -XX:+HeapDumpOnOutOfMemoryError OOM Dump

GC（垃圾回收器）

垃圾回收的区域只有在堆中和方法区

JVM在进行GC的时候，并不是堆这三个区域进行统一回收，大部分的时候都是在新生代

  • 新生代
  • 幸存区（from、to）
  • 老年区

GC两种类型：轻GC（普通GC）、重GC（全局GC）

常用算法

标记清除法

  • 优点：不需要额外的空间
  • 缺点：需要扫描两次，严重浪费时间，产出内存碎片

标记压缩

标记清除再优化

多了移动成本

复制算法（年轻代一般使用From和To）

每次GC都会清理Eden区活得对象移动到幸存区被GC后，就会是空的

也就是说每次GC过后幸存区活下来的对象都会是换一个幸存区，比如有幸存一区to和二区from，一区里面GC了之后有了两个幸存对象，那么一区就会和二区交换位置（复制算法从上到下），变成前幸存一区（现幸存二区），这时候二区就由from变成了to

  • 好处：没有内存碎片
  • 坏处：浪费一半的to区空间；但是假设有个对象100%存活（极端情况）

复制算法使用最佳场景：对象存货度较低的时候，新生区

引用计数法

总结

内存效率：复制算法 > 标记算法 > 标记压缩算法（时间复杂度）

内存整齐度：复制算法 = 标记压缩算法 > 标记清除算法

内存利用率：标记压缩算法 = 标记清除算法 > 复制算法
